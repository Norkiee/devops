# Sync — Figma to Azure DevOps Plugin

## Project Overview

Sync is a Figma plugin that allows UI/UX designers to convert design work directly into Azure DevOps work items with minimal manual effort. The plugin uses Claude AI to analyze selected frames or sections and generate **Epics**, **Features**, **User Stories**, or **Tasks**, dynamically adapting to your project's Azure DevOps hierarchy.

### Problem Statement

Designers create screens in Figma but must manually log work items in Azure DevOps afterward. Azure DevOps' work item creation is click-heavy, form-driven, and disruptive to creative workflow. This results in wasted time, context switching, and poor/delayed documentation.

### Solution

A Figma plugin that:
1. Reads selected frames or sections (names, text layers, component names)
2. Detects available work item types in your Azure DevOps project
3. Fetches Azure DevOps hierarchy for context (Epic, Feature, User Story)
4. Uses Claude AI to generate work item drafts (Epics, Features, User Stories, or Tasks)
5. Pushes work items to Azure DevOps with proper parent linking

### Target Users

- UI/UX Designers
- Product Designers
- Design teams working within Azure DevOps environments

## Azure DevOps Hierarchy

The plugin dynamically adapts to your project's work item types. Common hierarchies include:

**Agile Process:**
```
Epic (can be generated by plugin)
└── Feature (can be generated by plugin)
    └── User Story (can be generated by plugin)
        └── Task (can be generated by plugin)
```

**Scrum Process:**
```
Epic (can be generated by plugin)
└── Feature (can be generated by plugin)
    └── Product Backlog Item (treated as User Story)
        └── Task (can be generated by plugin)
```

**Basic Process:**
```
Epic (can be generated by plugin)
└── Issue (can be generated by plugin)
    └── Task (can be generated by plugin)
```

**Key points:**
- **Dynamic detection**: The plugin detects which work item types exist in your project
- **Only shows available types**: If your project doesn't have Features, the Feature option won't appear
- **Flexible parenting**: Epics have no parent; Features can optionally link to Epics; User Stories link to Epics or Features; Tasks link to User Stories
- The plugin reads the full hierarchy chain to provide AI with context

## User Flow

### Flow Order

```
1. Select frames/sections in Figma
2. Choose work item type: [ Epic ] [ Feature ] [ User Story ] [ Task ]
   (Only types available in your project are shown)
3. Connect to Azure DevOps (if not authenticated)
4. Select parent (dynamic based on type):
   - For Epics: Org → Project (no parent needed)
   - For Features: Org → Project → Epic (optional)
   - For User Stories: Org → Project → Epic → Feature (optional)
   - For Tasks: Org → Project → Epic → User Story
5. Add context (optional) — AI now has hierarchy context
6. Generate work items
7. Review, edit, select/deselect
8. Submit to Azure DevOps
```

## Data Models

### Frame Data (Plugin → Backend)

```typescript
interface FrameData {
  id: string;
  name: string;
  sectionName?: string;          // Which section this frame belongs to (if any)
  textContent: string[];         // Extracted text layers (max 30)
  componentNames: string[];      // Component instances used (max 20)
  nestedFrameNames: string[];    // Named child frames for structure context (max 10)
  width: number;
  height: number;
}
```

### Azure Hierarchy Context (Backend internal)

```typescript
interface HierarchyContext {
  epic?: {
    id: number;
    title: string;
    description?: string;
  };
  userStory?: {
    id: number;
    title: string;
    description?: string;
    acceptanceCriteria?: string;
  };
}
```

### Generate Request (Plugin → Backend)

```typescript
type WorkItemType = 'Epic' | 'Feature' | 'UserStory' | 'Task';

interface GenerateRequest {
  frames: FrameData[];
  context?: string;                    // Optional user-provided context
  workItemType: WorkItemType;          // What to generate
  hierarchyContext: HierarchyContext;  // Parent work items for AI context
}
```

### Generated Work Items (Backend → Plugin)

```typescript
interface WorkItem {
  id: string;                  // Unique ID for UI (e.g., "frame123-story1")
  title: string;
  description: string;
  acceptanceCriteria?: string; // Only for User Stories
  selected: boolean;           // User can deselect before submission
}

interface FrameWorkItems {
  frameId: string;
  frameName: string;
  sectionName?: string;
  workItems: WorkItem[];
}

interface GenerateResponse {
  workItemType: WorkItemType;
  frameWorkItems: FrameWorkItems[];
}
```

### Azure DevOps Work Items (Plugin → Backend → Azure)

```typescript
interface AzureUserStory {
  title: string;
  description: string;
  acceptanceCriteria?: string;
  parentEpicId: number;
  tags: string[];
  state: 'New';
}

interface AzureTask {
  title: string;
  description: string;
  parentStoryId: number;
  tags: string[];
  state: 'New';
  assignedTo?: string;  // Auto-assigned to current user
}

interface CreateWorkItemResult {
  workItemId: string;          // Our internal ID
  success: boolean;
  azureId?: number;            // Azure DevOps ID
  url?: string;                // Azure DevOps URL
  error?: string;
}
```

### Plugin Storage (Persisted)

```typescript
interface PluginStorage {
  azureOrg?: string;
  azureProjectId?: string;
  lastEpicId?: number;
  lastStoryId?: number;
  frequentTags?: string[];
  sessionId?: string;          // Server session ID for token refresh
  accessToken?: string;        // Stored for session persistence across plugin restarts
}
```

> **Note:** Both `accessToken` and `sessionId` are stored in plugin storage to persist the authentication session across plugin restarts. The `sessionId` is used to refresh expired tokens via the server.

## API Endpoints

### Backend Routes

#### `POST /api/generate`

Generate work item drafts (User Stories or Tasks) from frame data using Claude.

**Request (User Stories):**
```json
{
  "frames": [
    {
      "id": "123:456",
      "name": "Login Screen",
      "sectionName": "Onboarding",
      "textContent": ["Email", "Password", "Sign In", "Forgot password?"],
      "componentNames": ["Input", "Button", "Link"],
      "nestedFrameNames": ["Form Container", "Footer Links"],
      "width": 375,
      "height": 812
    }
  ],
  "context": "User authentication flow for mobile app",
  "workItemType": "UserStory",
  "hierarchyContext": {
    "epic": {
      "id": 100,
      "title": "User Onboarding",
      "description": "Enable new users to sign up, authenticate, and complete profile setup"
    }
  }
}
```

**Response (User Stories):**
```json
{
  "workItemType": "UserStory",
  "frameWorkItems": [
    {
      "frameId": "123:456",
      "frameName": "Login Screen",
      "sectionName": "Onboarding",
      "workItems": [
        {
          "id": "123:456-1",
          "title": "User can log in with email and password",
          "description": "As a user, I want to log in using my email and password so that I can access my account.",
          "acceptanceCriteria": "- Email field validates format\n- Password field masks input\n- Error messages display for invalid credentials\n- Successful login redirects to dashboard",
          "selected": true
        },
        {
          "id": "123:456-2",
          "title": "User can reset forgotten password",
          "description": "As a user, I want to reset my password if I forget it so that I can regain access to my account.",
          "acceptanceCriteria": "- Forgot password link is visible\n- Email input for reset request\n- Confirmation message after submission",
          "selected": true
        }
      ]
    }
  ]
}
```

**Request (Tasks):**
```json
{
  "frames": [...],
  "context": "User authentication flow for mobile app",
  "workItemType": "Task",
  "hierarchyContext": {
    "epic": {
      "id": 100,
      "title": "User Onboarding",
      "description": "Enable new users to sign up, authenticate, and complete profile setup"

      "id": 200,
      "title": "User can log in with email and password",

```

```json
{
  "workItemType": "Task",
  "frameWorkItems": [
    {
      "frameId": "123:456",
```

### api/_lib/redis.ts

Redis client wrapper using ioredis. Includes atomic `kvGetDel` to prevent race conditions.

```typescript
import Redis from 'ioredis';

let client: Redis | null = null;

function getClient(): Redis {
  if (!client) {
    const url = process.env.REDIS_URL;
    if (!url) {
      throw new Error('Missing REDIS_URL environment variable');
    }
    client = new Redis(url);
  }
  return client;
}

export async function kvGet<T>(key: string): Promise<T | null> {
  const redis = getClient();
  const value = await redis.get(key);
  if (!value) return null;
  try {
    return JSON.parse(value) as T;
  } catch (err) {
    console.error(`Failed to parse Redis value for key ${key}:`, err);
    return null;
  }
}

// Atomic get-and-delete to prevent race conditions
export async function kvGetDel<T>(key: string): Promise<T | null> {
  const redis = getClient();
  const value = await redis.getdel(key);
  if (!value) return null;
  try {
    return JSON.parse(value) as T;
  } catch (err) {
    console.error(`Failed to parse Redis value for key ${key}:`, err);
    return null;
  }
}

export async function kvSet(
  key: string,
  value: unknown,
  expirySeconds?: number
): Promise<void> {
  const redis = getClient();
  const serialized = JSON.stringify(value);
  if (expirySeconds) {
    await redis.set(key, serialized, 'EX', expirySeconds);
  } else {
    await redis.set(key, serialized);
  }
}

export async function kvDel(key: string): Promise<void> {
  const redis = getClient();
  await redis.del(key);
}
```

### api/_lib/claude.ts

Claude API wrapper with retry logic for transient errors (429, 503, 529).

```typescript
const MAX_RETRIES = 3;
const INITIAL_DELAY_MS = 1000;
const FETCH_TIMEOUT_MS = 45000; // 45 second timeout
const RETRYABLE_STATUS_CODES = [429, 503, 529];

async function fetchWithRetry(
  url: string,
  options: RequestInit,
  retries = MAX_RETRIES
): Promise<Response> {
  let lastError: Error | null = null;

  for (let attempt = 0; attempt <= retries; attempt++) {
    const controller = new AbortController();
    const timeoutId = setTimeout(() => controller.abort(), FETCH_TIMEOUT_MS);

    try {
      const response = await fetch(url, {
        ...options,
        signal: controller.signal,
      });
      clearTimeout(timeoutId);

      if (response.ok) return response;

      // Retry on transient errors with exponential backoff
      if (RETRYABLE_STATUS_CODES.includes(response.status) && attempt < retries) {
        const delay = INITIAL_DELAY_MS * Math.pow(2, attempt);
        await sleep(delay);
        continue;
      }

      const errorText = await response.text();
      lastError = new Error(`Anthropic API error (${response.status}): ${errorText}`);
      break;
    } catch (error) {
      clearTimeout(timeoutId);
      if (error instanceof Error && error.name === 'AbortError') {
        lastError = new Error(`Request timed out after ${FETCH_TIMEOUT_MS}ms`);
        if (attempt < retries) {
          const delay = INITIAL_DELAY_MS * Math.pow(2, attempt);
          await sleep(delay);
          continue;
        }
      } else {
        lastError = error instanceof Error ? error : new Error(String(error));
      }
      break;
    }
  }

  throw lastError || new Error('Unknown error during fetch');
}
```

```typescript
import { FrameData } from './types';

// Helper: Extract text content from frame (recursive)
function extractTextContent(node: SceneNode): string[] {
  const textContent: string[] = [];

  function traverse(n: SceneNode) {
    if (n.type === 'TEXT') {
      const text = n.characters.trim();
      if (text && text.length > 1 && !textContent.includes(text)) {
        textContent.push(text);
      }
    }
    if ('children' in n) {
      n.children.forEach(traverse);
    }
  }

  traverse(node);
  return textContent.slice(0, 30);
}

// Helper: Extract component instance names (recursive)
function extractComponentNames(node: SceneNode): string[] {
  const componentNames: string[] = [];

  function traverse(n: SceneNode) {
    if (n.type === 'INSTANCE') {
      const name = n.name;
      if (name && !name.match(/^(Frame|Group|Rectangle|Ellipse)\s*\d*$/i)) {
        if (!componentNames.includes(name)) {
          componentNames.push(name);
        }
      }
    }
    if ('children' in n) {
      n.children.forEach(traverse);
    }
  }

  traverse(node);
  return componentNames.slice(0, 20);
}

// Helper: Extract nested frame names
function extractNestedFrameNames(node: SceneNode): string[] {
  const frameNames: string[] = [];

  function traverse(n: SceneNode, depth: number) {
    if (depth > 2) return;
    if (n.type === 'FRAME' && n !== node) {
      const name = n.name;
      if (name && !name.match(/^Frame\s*\d*$/i)) {
        frameNames.push(name);
      }
    }
    if ('children' in n) {
      n.children.forEach(child => traverse(child, depth + 1));
    }
  }

  traverse(node, 0);
  return frameNames.slice(0, 10);
}

// Build frame data object
function buildFrameData(frame: FrameNode, sectionName?: string): FrameData {
  return {
    id: frame.id,
    name: frame.name,
    sectionName,
    textContent: extractTextContent(frame),
    componentNames: extractComponentNames(frame),
    nestedFrameNames: extractNestedFrameNames(frame),
    width: Math.round(frame.width),
    height: Math.round(frame.height),
  };
}

// Get frames from selection (supports both frames and sections)
function getFramesFromSelection(): { frames: FrameData[]; sections: string[] } {
  const frames: FrameData[] = [];
  const sections: string[] = [];

  for (const node of figma.currentPage.selection) {
    if (node.type === 'FRAME') {
      // Direct frame selection
      frames.push(buildFrameData(node));
    } else if (node.type === 'SECTION') {
      // Section selected — get all child frames
      sections.push(node.name);
      const sectionFrames = node.children
        .filter((child): child is FrameNode => child.type === 'FRAME')
        .map(frame => buildFrameData(frame, node.name));
      frames.push(...sectionFrames);
    }
  }

  return { frames, sections };
}

// Listen for UI requests
```

## Azure DevOps Integration

### Authentication: Microsoft Entra ID OAuth

Azure DevOps OAuth is deprecated (April 2025) and will be removed in 2026. New apps must use **Microsoft Entra ID OAuth**.

> **Note:** Microsoft Entra ID OAuth does not natively support personal Microsoft accounts (MSA) for Azure DevOps. It works with organizational accounts (Azure AD backed). If your team uses personal Microsoft accounts, users may need to be added to your Azure DevOps organization with work accounts.

#### App Registration (Azure Portal)

1. Go to [Azure Portal](https://portal.azure.com) → Microsoft Entra ID → App registrations → New registration
2. Name: "Sync Figma Plugin"
3. Supported account types: "Accounts in any organizational directory" (for multi-tenant)
4. Redirect URI: Web → `https://your-project.vercel.app/api/azure/callback`
5. Click Register

#### Configure API Permissions

1. Go to your app → API permissions → Add a permission
2. Select "Azure DevOps" (not Microsoft Graph)
3. Select Delegated permissions:
   - `user_impersonation` (Access Azure DevOps resources)
4. Click "Grant admin consent" (if you have admin rights, otherwise users consent on first login)

#### Create Client Secret

1. Go to Certificates & secrets → New client secret
2. Description: "Vercel backend"
3. Expiry: 24 months (set a reminder to rotate)
4. Copy the secret value immediately (shown only once)

#### Key Configuration Values

| Value | Where to find it |
|-------|------------------|
| Client ID | App registration → Overview → Application (client) ID |
| Client Secret | Certificates & secrets → Value (copy when created) |
| Tenant ID | App registration → Overview → Directory (tenant) ID |
| Azure DevOps Resource ID | `499b84ac-1321-427f-aa17-267ca6975798` (constant) |

### OAuth Flow (Polling-Based)

The plugin uses a **polling-based OAuth flow** instead of `window.postMessage` to avoid cross-origin issues in Figma's plugin environment.

**Flow:**
```
1. Plugin generates unique state UUID
2. Plugin opens popup: GET /api/azure/auth?state={state}
3. User authenticates with Azure
4. Azure redirects to: GET /api/azure/callback?code=...&state={state}
5. Callback exchanges code for tokens
6. Callback stores result in Redis: auth:{state} (5 min TTL)
7. Callback shows styled success page with animated checkmark
8. Plugin polls: GET /api/azure/poll?state={state}
9. Poll returns { status: 'pending' } or { status: 'complete', sessionId, accessToken }
10. Plugin stores sessionId for token refresh, accessToken in memory
```

## Development Guidelines

### Code Style

- TypeScript strict mode enabled
- Functional React components with hooks
- Named exports (no default exports except pages)
- Explicit return types on functions
- Use `interface` over `type` for object shapes

### Error Handling

- All API calls wrapped in try/catch
- User-friendly error messages (never expose raw errors)
- Retry logic for network failures (max 3 attempts)
- Graceful degradation when Azure API is slow

### Security

- Access tokens stored in memory (React state), never in plugin storage
- Refresh tokens stored server-side in Redis, never exposed to client
- Session IDs (not tokens) stored in plugin storage for token refresh
- API keys only in backend environment variables
- Validate all inputs on backend before processing
- Rate limit API endpoints

### Testing Priorities

1. Frame data extraction accuracy (text content, component names, nested frames)
2. Section selection and child frame extraction
3. Claude API retry logic (429, 503, 529 errors, timeouts)
4. Azure OAuth flow (happy path + token refresh + session persistence)
5. Atomic polling (kvGetDel prevents duplicate token consumption)
6. User Story creation with acceptance criteria
7. Task creation with parent linking
8. Auto-assignment of tasks to current user
9. Partial failure and retry logic
10. Plugin auto-resize for dynamic content

### Deploy API to Vercel

**Recommended: Git-based deployment (auto-deploy)**

The project is connected to GitHub for automatic deployments. Push to `main` branch to trigger a production deployment:

```bash
git add .
git commit -m "Your commit message"
git push
```

Vercel will automatically build and deploy when changes are pushed to the `main` branch.

**Alternative: Manual deployment via CLI**

```bash
# Install Vercel CLI globally
npm install -g vercel

# Login to Vercel
vercel login

# Link to new Vercel project
vercel link

# Set environment variables in Vercel dashboard:
# - ANTHROPIC_API_KEY
# - AZURE_CLIENT_ID
# - AZURE_CLIENT_SECRET
# - AZURE_REDIRECT_URI (https://your-project.vercel.app/api/azure/callback)

# Deploy
vercel --prod
```

### Set Up Redis Storage

You can use any Redis provider:

**Option 1: Upstash (recommended for Vercel)**
1. Go to [Upstash](https://upstash.com)
2. Create a new Redis database
3. Copy the `REDIS_URL` connection string
4. Add to Vercel environment variables

**Option 2: Redis Cloud**
1. Go to [Redis Cloud](https://redis.com/try-free/)
2. Create a free database
3. Get the connection URL
4. Add to Vercel environment variables

**Option 3: Self-hosted**
1. Run Redis locally or on a server
2. Use the connection URL format: `redis://username:password@host:port`
